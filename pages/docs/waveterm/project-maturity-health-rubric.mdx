# Project Maturity and Health Rubric

**Version:** 1.0  
**Scale:** 1-10 per criterion  
**Target markers:** ~500 (SpaceX/Google-level)  
**Scope:** Any project in workspace (see `.vscode/sessions.json`).

---

## Purpose and scope

- **Purpose:** Assess maturity and health of any software project in the Brightpath workspace using
  a single, project-agnostic rubric.
- **Scope:** All projects referenced in `.vscode/sessions.json` (kid_omega, avengers,
  brightpath_swarm, brightpath_cli, zudoku, PathX_Matcher, wireweave, chartdb, etc.) and any future
  repo.
- **Use:** Score each criterion 1-10; mark N/A where the dimension does not apply; record evidence;
  compute roll-up (average of applicable criteria or weighted). Re-assess periodically (e.g.
  quarterly) or on major releases.
- **N/A rules:** No Flutter → Patrol (2.7), Maestro (2.8) N/A. No auth → 8.3 N/A. No BDD → 2.9 N/A.
  Single-tenant → 11.4 N/A. Minimal app → 6.3 tracing N/A. Mark "N/A" in scorecard and exclude from
  roll-up.
- **Evidence:** Each criterion has evidence prompts and markers. Evidence should be in repo, CI, or
  docs (version control or audit log preferred over spreadsheets for durability and reversibility of
  assessment).

## Section 0: 160-IQ-style analytical dimensions (~55 markers)

High-level analytical rigor: first-principles, systems thinking, causal reasoning, failure-mode
rigor, trade-off explicitness. _Ref: SpaceX design principles, Google SRE, first-principles
reasoning._

### 0.1 First-principles and problem decomposition (1-10)

**Definition:** The problem is defined from first principles; success and anti-goals are explicit;
constraints and assumptions are written and validated; the solution is decomposed into verifiable
sub-problems with traceability.

**Level descriptions:**

- **1:** No written problem statement or success criteria; work is demand-driven only.
- **3:** Short problem statement exists; some success criteria; constraints or assumptions are
  implicit.
- **5:** Problem statement, success criteria, and anti-goals documented; constraints doc or ADR;
  assumption list present with validation cadence.
- **7:** Decomposition into sub-problems with acceptance criteria; traceability to requirements or
  tickets; assumptions validated on a schedule.
- **9:** First-principles breakdown is the source of truth for scope; invertibility: "If we removed
  X, would the problem statement still hold?" is answerable.

**Evidence prompts:** Look for problem statement or vision doc; success/anti-goals; constraints doc
or ADR; assumption register; breakdown structure (e.g. epic/feature list) with acceptance criteria.

**Markers (11):**

1. Written problem statement exists.
2. Success criteria are explicit and measurable.
3. Anti-goals (out of scope) are stated.
4. Constraints (time, cost, tech, compliance) are documented.
5. Assumption list exists.
6. Assumptions have a validation cadence.
7. Breakdown structure (sub-problems or work items) exists.
8. Acceptance criteria exist per sub-problem or feature.
9. Traceability from work to problem statement (or requirements).
10. _Invertibility:_ Evidence that the project would fail the bar if the problem were not addressed
    from first principles (e.g. no ceremony-only docs).
11. Reversibility of assessment: evidence is in version control or audit log.

---

### 0.2 Systems thinking and boundaries (~11 markers)

**Definition:** System boundary (in-scope vs external) is explicit; dependencies and failure
propagation are mapped; feedback loops (reinforcing/balancing) are considered; emergent behavior at
scale is considered.

**Level descriptions:**

- **1:** No boundary or dependency documentation.
- **3:** Simple context diagram or list of integrations; no failure propagation.
- **5:** Context diagram and boundary doc; dependency map; failure-impact matrix or narrative.
- **7:** Feedback loops (e.g. load → errors → retries → load) documented or diagrammed; load/chaos
  hypotheses for emergent behavior.
- **9:** System is analyzed as a set of loops and boundaries; unknown-unknowns process (chaos, red
  team, tabletop) feeds back into design.

**Evidence prompts:** Context diagram; boundary doc; interface inventory; dependency map;
failure-impact matrix; loop diagram or narrative; load/chaos hypotheses.

**Markers (11):**

1. Context diagram exists.
2. Boundary (in-scope vs external) is documented.
3. Interface inventory (APIs, events, queues) exists.
4. Dependency map (upstream/downstream) exists.
5. Failure-impact matrix or equivalent (when X fails, Y is affected) exists.
6. Feedback loops (reinforcing/balancing) are identified.
7. Emergent behavior at scale is considered (document or test).
8. Load or chaos hypotheses exist.
9. _Invertibility:_ No critical path is undocumented (negative marker: fail if critical path has no
   failure scenario).
10. Process to discover new failure modes (chaos, game-day, tabletop) exists and feeds runbooks.
11. Evidence durable (version control or audit log).

---

### 0.3 Causal reasoning and invariants (~11 markers)

**Definition:** Causal model for critical behaviors is explicit; invariants (e.g. balance, ordering)
are documented and tested; counterfactual and rollback reasoning are applied.

**Level descriptions:**

- **1:** No causal or invariant documentation.
- **3:** Some "when X then Y" in runbooks or comments; no invariant tests.
- **5:** Causal diagram or narrative for critical paths; invariant list; property or invariant tests
  for key invariants.
- **7:** Rollback scenarios documented and tested; postmortems linked to causal model and rubric
  dimensions.
- **9:** Causal model is the basis for incident analysis; "If this runbook were wrong, how would we
  discover it?" is answerable.

**Evidence prompts:** Causal diagram or narrative; invariant list; property/invariant tests;
rollback scenarios; postmortem linkage to rubric.

**Markers (11):**

1. Causal diagram or narrative for critical behaviors exists.
2. Invariant list (e.g. balance non-negative, ordering) exists.
3. Property-based or invariant tests exist for key invariants.
4. Rollback scenarios are documented.
5. Rollback is tested (e.g. in drill or CI).
6. Postmortems reference causal model or root cause.
7. Postmortems are blameless and link to rubric dimensions (e.g. "maps to 9.3 Chaos").
8. _Counterfactual:_ "If runbook were wrong, how would we discover it?" is documented or
   drill-based.
9. No critical path without a causal or rollback story (negative marker).
10. Reversibility of assessment: evidence durable.
11. Metrics used are well-defined, measurable, and actionable (would change a decision).

---

### 0.4 Failure-mode rigor (FMEA-light) (~12 markers)

**Definition:** Failure modes for critical paths are enumerated; severity and mitigation or accepted
risk are explicit; SPOFs are identified and mitigated or accepted; recovery and repair are defined
and practiced.

**Level descriptions:**

- **1:** No failure-mode analysis.
- **3:** Informal list of "what can go wrong"; no severity or mitigation.
- **5:** FMEA table or failure-mode list for critical paths; severity/criticality; SPOF list with
  mitigation or accepted risk.
- **7:** Failure-scenario docs; recovery runbooks; repair SLAs; drill cadence.
- **9:** Unknown-unknowns process (chaos, red team) updates failure modes and runbooks; no
  unmitigated SPOF on critical path.

**Evidence prompts:** FMEA table or failure-mode list; severity/criticality; SPOF list;
failure-scenario docs; recovery runbooks; repair SLAs; drill cadence.

**Markers (12):**

1. FMEA table or failure-mode list exists for at least one critical path (deploy, data, auth,
   payment, etc.).
2. Severity (or equivalent) is assigned per failure mode.
3. Likelihood or detectability (or equivalent) is considered.
4. Mitigation or accepted risk is explicit per failure mode.
5. SPOF list exists.
6. Each SPOF has mitigation or accepted risk documented.
7. Failure-scenario docs (when X fails, Y happens) exist.
8. Recovery runbooks exist and are linked to failure modes.
9. Repair SLAs or targets exist where relevant.
10. Drill cadence (e.g. game-day, restore test) is defined and used.
11. _Invertibility:_ No critical path has unmitigated SPOF (negative marker).
12. Process to discover new failure modes and update runbooks exists.

### 0.5 Trade-off explicitness (~10 markers)

**Definition:** Major trade-offs (e.g. consistency vs availability, cost vs reliability) are written
with rationale; reversibility (one-way vs two-way door) is classified; options are compared on
shared dimensions.

**Level descriptions:**

- **1:** No trade-off documentation.
- **3:** Some trade-offs mentioned in ADRs or comments; no option table.
- **5:** ADRs or trade-off log with rationale; reversibility tag (one-way vs two-way) on major
  decisions; option table for at least one major decision.
- **7:** Trade-offs are the basis for design review; option tables with criteria and scoring;
  "reversibility of assessment" (evidence in version control) so re-run gives comparable result.
- **9:** All material trade-offs are documented; decision quality is reviewed; assessment evidence
  is durable and re-runnable.

**Evidence prompts:** ADRs or trade-off log; reversibility tag; option table with criteria and
scoring; evidence in version control.

**Markers (10):**

1. ADRs or trade-off log exists.
2. Major decisions include rationale.
3. Reversibility (one-way vs two-way door) is tagged where relevant.
4. Option table (alternatives compared) exists for at least one major decision.
5. Comparison criteria are explicit.
6. Trade-offs are referenced in design or architecture docs.
7. Assessment evidence is in version control or audit log (reversibility of assessment).
8. Re-running the rubric would yield comparable result (process or tooling).
9. No major trade-off left implicit for critical path (negative marker).
10. Human/organizational: rubric owner and review cadence are defined.

## Section 1: Code quality and structure (~35 markers)

### 1.1 Layering and separation of concerns (1-10)

**Definition:** Code is organized in clear layers (e.g. api/domain/infra or ui/services/data) with
explicit boundaries; no circular dependencies; internal boundaries are enforced (e.g. via packages
or modules).

**Level descriptions:** 1 = single blob; 3 = some split; 5 = clear layers; 7 = bounded contexts or
modules; 9 = DDD-style or documented modular architecture with dependency rules.

**Evidence:** Directory layout; presence of api/domain/infra or equivalent; internal boundaries;
dependency direction (e.g. no domain importing from api).

**Markers (8):** Layered directories; api/domain/infra (or equivalent); no circular deps; boundary
doc or convention; dependency rule enforced (linter or CI); _invertibility:_ no critical domain
logic in presentation layer; evidence in repo.

---

### 1.2 Naming and conventions (1-10)

**Definition:** Naming is consistent with a style guide; linter enforces conventions; test names
follow a pattern (e.g. TestX_Scenario_Expected).

**Level descriptions:** 1 = inconsistent; 3 = partial; 5 = style guide and linter; 7 = documented
and mostly followed; 9 = zero known violations; exceptions documented.

**Evidence:** Lint config; naming samples; test name pattern.

**Markers (6):** Style guide or convention doc; linter for naming; test naming pattern; no
abbreviations in public API (or documented); evidence in repo.

---

### 1.3 Readability and complexity (1-10)

**Definition:** Functions are reasonably short (< ~40 lines); cyclomatic complexity < 10 (or
budget); nesting limited; control flow is clear.

**Level descriptions:** 1 = many long/complex functions; 3 = some tooling; 5 = complexity budget and
tool (gocyclo, radon, ESLint); 7 = enforced in CI; 9 = trended and refactored when over budget.

**Evidence:** Tool output (gocyclo, radon, ESLint); comments for complex logic.

**Markers (6):** Complexity tool in use; budget (e.g. < 10) defined; CI gate; no function > 40 lines
without exception doc; _metric that would change a decision_ (e.g. refactor when > 15); evidence in
repo.

---

### 1.4 Lint, format, and type safety (1-10)

**Definition:** Linters and formatters are configured and run in CI; strict type mode where
applicable; zero known violations or documented exceptions.

**Level descriptions:** 1 = none; 3 = local only; 5 = CI; 7 = strict mode; 9 = zero violations;
exceptions documented.

**Evidence:** Config files; CI steps; zero known violations.

**Markers (6):** Lint config; format config; type strict (TS strict, mypy strict, etc.); CI runs
lint/format/type; zero violations or exception list; evidence in repo.
