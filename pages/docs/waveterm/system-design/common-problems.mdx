## URL Shortener

**Requirements:** URL shortening service like bit.ly

**Scale Estimate:** 10M DAU, 100M requests/day, read/write ratio 100:1

**System Design:**

- High-level: Application servers + database + cache
- Database choices: SQL for relational queries, NoSQL for scale
- 62^7 keys (~3.5T possible, base62 = [a-zA-Z0-9])

**Scaling:**

- Database sharding by hash of user ID
- 6-8 character keys
- CDN for redirect optimization
- Analytics and click tracking

**Failure Modes:** Cache misses, database bottlenecks, single point of failure in ID generation

## Pastebin

**Requirements:** Text sharing service like pastebin.com

**Scale Estimate:** 1M DAU, 10M pastes/month, average paste size 10KB

**System Design:**

- High-level: Load balancer + application servers + database + blob storage
- Content deduplication for space efficiency
- Key generation: 62^n keys, sharing via short URLs

**Scaling:**

- Database for metadata (MySQL/PostgreSQL)
- Blob storage for content (S3, GCS)
- Caching (Redis) for popular pastes
- CDN for global distribution

**Failure Modes:** Data loss, blob storage unavailability, key collision

## Rate Limiter

**Requirements:** Prevent system abuse by limiting requests

**Scale Estimate:** 10M DAU, 1B requests/day

**System Design:**

- Algorithms: Token bucket, leaky bucket, fixed window, sliding window
- Counter-based vs distributed (Redis)
- Multi-tier limits: per user/hour, per user/day, global limits

**Scaling:**

- In-memory data structures (Redis sorted sets)
- Distributed rate limiting
- Consistent hashing for fairness

**Failure Modes:** Redis single point of failure, clock skew in distributed systems

## Key-Value Store

**Requirements:** High-performance key-value storage

**Scale Estimate:** 1M qps writes, 10M qps reads, 100TB data

**System Design:**

- Consistent hashing for data distribution
- Replication for fault tolerance
- CAP tradeoffs: AP systems (Dynamo), CP systems (Cassandra)

**Scaling:**

- Horizontal scaling with partitioning
- Read replicas for read-heavy loads
- Compression and tiered storage

**Failure Modes:** Network partitions, data inconsistency, node failures

## Distributed Cache

**Requirements:** Faster data access than database

**Scale Estimate:** 100M qps, 1TB data, <1ms latency

**System Design:**

- Cache strategies: LRU, LFU, TTL
- Architecture: Application cache + distributed cache
- Invalidation: Write-through, write-behind, refresh-ahead

**Scaling:**

- Memcached/Redis clusters
- Consistent hashing
- Multi-level cache hierarchy

**Failure Modes:** Cache stampede, hot keys, cold starts

## Search Autocomplete

**Requirements:** Type-ahead search suggestions

**Scale Estimate:** 10M DAU, 100M queries/day

**System Design:**

- Data structure: Trie (prefix tree)
- Top-k results per prefix
- Real-time vs batch indexing

**Scaling:**

- Trie sharding by first character
- Redis for fast lookups
- Analytics-driven ranking

**Failure Modes:** High cardinality prefixes, memory limits

## Web Crawler

**Requirements:** Systematic web crawling

**Scale Estimate:** 1B pages, 1M qps

**System Design:**

- Components: Scheduler, fetchers, processors
- Politeness: Robots.txt, rate limiting, backoff
- URL frontier: politeness queues

**Scaling:**

- Distributed crawling (thousands of machines)
- URL deduplication (Bloom filters)
- Freshness tracking

**Failure Modes:** Site quotas, duplicate content, infinite loops

## Duplicate File Detection

**Requirements:** Efficient duplicate detection in storage systems

**Scale Estimate:** 1B files, real-time detection

**System Design:**

- Fingerprinting: SHA-256, MD5, content checksums
- Bloom filters for exists checks
- Rolling hashes for near-duplicate detection

**Scaling:**

- Distributed hashing
- Hierarchical storage management
- Similarity search (shingling, MinHash)

**Failure Modes:** Hash collisions, false positives/negatives

## News Feed

**Requirements:** Social media feed generation

**Scale Estimate:** 100M users, 1B events/day

**System Design:**

- Fanout-on-write vs fanout-on-read
- Timeline tables vs materialized views
- Push vs pull models

**Scaling:**

- Sharding by user ID
- Redis for freshness
- Best-effort delivery

**Failure Modes:** Content moderation delays, high fanouts
